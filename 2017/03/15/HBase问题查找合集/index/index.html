HBase问题查找合集

Mar 15, 2017 | Hits

记录在运维HBase过程中遇到的问题和排查过程。

案件一：机器Active RPC 达到上限200

  • 起因

    某天晚上，收到集群报警短信——ActiveRPC 连续3次大于200，马上查看监控，发现确实也打满了。

  • 查找

    查看关键日志

    注意下面日志：

    memstore 居然超过了256 * hbase.hregion.memstore.block.multiplier = 768 ？

    理论上来说memstore达到设定的值（例如我们是256M）就flush到DN，当然整个的flush是一个异步过程，到底什么原因导致flush不了呢？

    再仔细看一下日志：

    出现上面日志的原因是HFile过多，超过了设置的hbase.hstore.blockingStoreFiles = 30；一旦超过这个限制，flush需要等待compaction完成之后，hfile减少了，才可以触发flush。

    查看监控信息验证自己想法：

    到了30分的日志，还继续打印HFile太多：

    那么问题来了：
    为什么HFile会瞬时这么多？
    compaction 不过来吗？

    查看监控这个时间点的compaction队列情况：

    确认队列积压
    查看日志，统计一下到底产生HFile的个数
    grep “ “ | awk -t

    发现在27分flush了52次，在29分钟flush了39次

    总结就是产生的HFile过多，同时compaction不过来，导致写阻塞了，从而导致handle打满。

  • 解决总结

    首先看一下业务是否存在热点写问题，是否可以提前做预分区；如果内存比较足，适当设大表的memstore大小，或者调整hbase.hregion.memstore.block.multiplier大小；适当增加compaction的线程个数；适当调大 hbase.hstore.blockingStoreFiles ，例如100。

案件2 java.lang.OutOfMemoryError: Requested array size exceeds VM limit

使用版本CDH5.4.5，HBase1.0.0

刚来新公司不久就出现RegionServer宕机事件，报的异常如下：

重点关注：“ Requested array size exceeds VM limit”

在 openjdk里面有限制数组的大小为2的31次方-2 ，如果超过这个大小，jvm就会报错。

这里其实可以肯定是HBase的ipc里面的一个bug，在某些情况下，创建的数组长度超过jvm的限制，通过搜索，找到了一个patch，修复就是这个问题。HBASE-14598) ，主要修改数组的长度，如果超过这个就会直接发送异常给客户端。直接原因就是，但是对于一个运维来说，其实更想知道到底是那张表的请求导致这个问题？

我们有打了一个patch，HBASE-16033) ，提供更多的log，最后找到了类型下面的日志：

这里主要是responsesize，就是一次返回的数据量太大了导致了这个问题。

另外，在搜索过程中，也找到有人出现了类似问题，类似，基本跟我们的类型。值得注意的是两个patch是：HBASE-14946 和 HBASE-14978，解决批量读写超过限制的问题。上面的pathc是解决不要报错问题，下面的才是根据。
